    23	## The Pipeline Mindset
    24	
    25	**Experiments fail for two reasons: wrong order and no stopping criteria.** Most researchers jump straight to testing their novel method without verifying their baseline setup, then wonder why results don't make sense. Others spend weeks tuning hyperparameters without a budget, hoping the next run will work.
    26	
    27	The 4-stage pipeline solves both problems. It enforces a strict order (each stage validates assumptions the next stage depends on) and assigns attempt budgets (forcing systematic thinking over brute-force iteration).

    40	## 4-Stage Pipeline Overview
    41	
    42	Each stage follows a **generate → execute → record → diagnose → revise** loop:
    43	
    44	| Stage | Goal | Budget (N_E^s) | Gate Condition |
    45	|-------|------|--------|----------------|
    46	| 1. Initial Implementation | Get baseline code running and reproduce known results | ≤20 attempts | Metrics within 2% of reported values (or within reported variance) |
    47	| 2. Hyperparameter Tuning | Optimize config for your setup | ≤12 attempts | Stable config, variance < 5% across 3 runs |
    48	| 3. Proposed Method | Implement & validate novel method | ≤12 attempts | Outperforms tuned baseline on primary metric, consistent across 3 runs |
    49	| 4. Ablation Study | Prove each component's contribution | ≤18 attempts | All claims evidenced with controlled experiments |
    50	
    51	Each stage saves artifacts to `/experiments/stageN_name/`.
    52	
    53	### The Stage Loop
    54	
    55	Within every stage, repeat this cycle for each attempt:
    56	
    57	1. **Generate**: Form a hypothesis or plan for this attempt. What specifically will you try? What do you expect to happen?
    58	2. **Execute**: Run the experiment. Record exact configuration, code changes, and runtime.
    59	3. **Record**: Log results immediately using the stage log template. Include both metrics and observations.
    60	4. **Diagnose**: Compare results to expectations. If they match, assess the gate condition. If they don't, load `experiment-craft` for the 5-step diagnostic flow.
    61	5. **Revise**: Based on diagnosis, either advance to the next stage (gate met) or plan the next attempt (gate not met).

    63	## Stage 1: Initial Implementation
    64	
    65	**Goal**: Find or generate executable baseline code and verify it reproduces published results. This stage corresponds to the paper's "initial implementation" — the engineer agent searches for working code, runs it, and records structured execution results.
    66	
    67	**Why this matters**: If you can't get the baseline running and reproducing known results, every subsequent comparison is meaningless. Initial implementation validates your data pipeline, evaluation code, training infrastructure, and understanding of prior work.
    68	
    69	**Budget**: ≤20 attempts (N_E^1=20). Baselines can be tricky — missing details in papers, version mismatches, unreported preprocessing steps. 20 attempts gives enough room to debug without allowing infinite tinkering.
    70	
    71	**Gate**: Primary metrics within 2% of reported values (or within the reported variance if provided).
    72	
    73	**Process**:
    74	1. Find the original baseline code (official repo, re-implementations, or write from paper description)
    75	2. Get the code running in your environment — resolve dependencies, fix compatibility issues
    76	3. Match the exact training configuration from the paper (dataset splits, preprocessing, hyperparameters)
    77	4. Run and compare metrics. If off by >2%, diagnose the gap
    78	5. Common pitfalls: different random seeds, different data splits, unreported data augmentation, framework version differences
    79	
    80	**When to load `experiment-craft`**: If attempts 1-5 all fail significantly (>10% gap), switch to the 5-step diagnostic flow to isolate the cause before burning more attempts.
    81	
    82	**Output**: `/experiments/stage1_baseline/` containing results, config, and verified baseline code.
    83	
    84	See [references/stage-protocols.md](references/stage-protocols.md) for detailed initial implementation checklists.
    85	
    86	## Stage 2: Hyperparameter Tuning
    87	
    88	**Goal**: Find the optimal hyperparameter configuration for YOUR specific setup.
    89	
    90	**Why this matters**: Published hyperparameters are tuned for the authors' setup. Your hardware, data version, framework version, or subtle implementation differences mean their config may not be optimal for you. Tuning now prevents confounding your novel method's results with suboptimal baselines.
    91	
    92	**Budget**: ≤12 attempts. Hyperparameter tuning has diminishing returns. If 12 structured attempts don't find a stable config, the problem is likely deeper than hyperparameters.
    93	
    94	**Gate**: Stable configuration found — variance < 5% across 3 independent runs with different random seeds.
    95	
    96	**Process**:
    97	1. Identify the most sensitive hyperparameters (usually: learning rate, batch size, loss weights)
    98	2. Start with coarse search on the most sensitive parameter
    99	3. Narrow the range based on results, then move to the next parameter
   100	4. Validate final config with 3 independent runs

   140	## Stage 4: Ablation Study
   141	
   142	**Goal**: Prove that each component of your method contributes meaningfully to the final result.
   143	
   144	**Why this matters**: Reviewers will ask "is component X really necessary?" for every part of your method. Without ablation, you can't answer. More importantly, ablation helps YOU understand why your method works — sometimes components you thought were important aren't, and vice versa.
   145	
   146	**Budget**: ≤18 attempts. Ablation requires multiple controlled experiments — one per component being ablated, plus interaction effects. 18 attempts covers a method with 4-5 components.
   147	
   148	**Gate**: Every claimed contribution is supported by a controlled experiment showing its effect.
   149	
   150	**Process**:
   151	1. List all components of your method that you claim contribute to performance
   152	2. Design ablation experiments: remove ONE component at a time, measure the impact
   153	3. For components that interact, test interaction effects
   154	4. Verify that no single component's removal improves results (would invalidate the claim)
   155	
   156	**Three ablation designs**:
   157	- **Leave-one-out**: Remove each component individually. Shows each component's marginal contribution.
   158	- **Additive**: Start from baseline, add components one at a time. Shows incremental gains.
   159	- **Substitution**: Replace your component with an alternative approach. Shows your component is better than alternatives, not just better than nothing.
   160	
   161	**When to load `experiment-craft`**: If ablation results contradict your hypothesis (removing a component improves results), use diagnostic flow to understand why.
   162	
   163	**Output**: `/experiments/stage4_ablation/` containing ablation results table, per-component analysis.
